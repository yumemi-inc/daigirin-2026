# EC2で1500万行のデータを1日で検証する

読者の皆さん初めまして、yuuukaです。PyLadies TokyoというPythonistaの女性コミュニティのスタッフをしています。今回はご縁でお声がけいただき、「ゆめみ大技林」に参加することができました。非常に嬉しいです。どのようなテーマで書くか（真面目系にするか、ネタに走るか）悩んだのですが、まずはPython関連で真面目に書いてみることにしました。今後また機会があったら色々挑戦したいと思います。

## 背景 — なぜ「ただのファイル比較」では済まなかったのか

### 発端

私は普段の業務でもPythonを使用しており、大規模なデータの分析、集計、可視化などを行っています。

詳細は省きますが、出力されるデータが完全に一致しているかを検証する必要がありました。対象は数十GBに及ぶTSV（タブ区切り）ファイルです。当初は「差分なんて`diff`コマンドか、Pythonでスクリプトを書けば終わるだろう」と思っていたのですが、実際のデータには3つの厄介な特性があり、幾度となく処理がクラッシュする事態に陥りました。
本稿では、業務で経験した数十GBのTSVファイルの整合性検証を背景に、EC2インスタンス上で環境構築からベンチマーク実行までを1日でやり切った話をまとめました。Pandas、Polars、DuckDBといったライブラリで1,500万行のデータをそれぞれ処理してみた結果、何が問題になって、最終的にどういう技術選定に至ったのかを共有したいと思います。

最初の検証は、手元のMacBook Pro（M3 Pro / メモリ36GB）で行いました。

**実業務のデータ規模**：

| 項目 | 値 |
| --- | --- |
| 行数 | 1,500万行 |
| ファイル数 | 100ファイル |
| ファイルサイズ（gzip圧縮） | 130MB/ファイル |
| ファイルサイズ（解凍後） | 850MB/ファイル |
| 比較対象 | BEFORE / AFTER の×2 |
| 合計処理対象 | 解凍後170GB相当 |

なお、後述するEC2での再検証では、同じ1,500万行・100ファイル構成のサンプルデータ（2.6GB）を生成して使用しています。

### データ仕様と「3つの罠」

検証対象のデータスキーマは以下の通りです。

| 列番号 | 物理名 | データ例 | 検証の条件 |
| --- | --- | --- | --- |
| 1 | `user_id` | `u.123456` | 主キー。プレフィックスを含む文字列。完全一致すること。 |
| 2 | `tags` | `101\|202` | 数値のパイプ区切りリスト。**順不同で一致**すること。 |
| 3 | `timestamps` | `2026-01-07 09:00:00` | 日時リスト。**文字列フォーマットも含めて完全一致**すること。 |
| 4 | `op_type` | `I`, `U`, `D` | 処理フラグ。`I`は`U`に読み替え、`D`は除外する。 |

#### 罠1：行とリスト要素の「順不同性」

分散処理システムから出力されたファイルなので、行の出力順序は毎回バラバラです。さらに厄介なことに、第2列・第3列の「パイプ区切りのリスト」も内部の順序が保証されていませんでした。つまり、旧システムが`101|202`と出力し、新システムが`202|101`と出力した場合、これらは「論理的に一致している」と判定する正規化処理を自前で実装する必要がありました。

#### 罠2：メモリの枯渇（Out of Memory）

数十GBのテキストデータをすべてメモリ上に展開してソートや結合を行おうとすると、瞬く間にRAMを食いつぶし、プロセスが強制終了されてしまいます。処理途中で幾度となくVSCodeやChromeなどが固まり、強制終了せざるを得なくなりました。

#### 罠3：「型推論」が引き起こす偽の一致

個人的にはこの罠が一番恐ろしいなと思いました。read_csv関数を使用すると、CSV/TSVを読み込む際に自動でデータ型を推論してくれるのですが、厳密なファイル比較においてはこれが深刻な問題を引き起こします。

- **前ゼロの消失**: 旧`"0123"`と新`"123"`。INT型として解釈されると両者とも`123`となり、一致と誤判定されます。
- **日時フォーマットの吸収**: 旧`"2026-01-07 09:00:00"`と新`"2026-01-07T00:00+0900"`。TIMESTAMP型としてパースされると、指し示す時刻が同じため一致と誤判定されます。

ファイル出力のテストである以上、文字列表記の差異は「不一致（バグ）」として検知しなければなりません。つまり、「絶対に型推論をさせず、すべてを文字列（VARCHAR）として比較する」という強い制約が必要でした。

## ローカルでの試行 — 各ライブラリの実装と限界

この要件を満たすために、ローカル環境（MacBook Pro M3 Pro / メモリ36GB）でPythonエコシステムの主要なツールを順番に試してみました。

### Pandas：小〜中規模データの王道

最初に試したのはPandasです。Pythonでのデータ分析といえばPandasですね。メモリ不足を回避するため、`chunksize`を用いて分割読み込みを行い、ハッシュ値で比較するアプローチをとりました。

```python
import pandas as pd
import hashlib
from collections import Counter

chunk_iter = pd.read_csv("before.tsv", sep="\t", dtype=str, chunksize=100000)
before_hashes = Counter()

def normalize_and_hash(row):
    if row['op_type'] == 'D': return None
    tags = "|".join(sorted(str(row['tags']).split("|")))
    ts = "|".join(sorted(str(row['timestamps']).split("|")))
    op = 'U' if row['op_type'] == 'I' else row['op_type']
    row_str = f"{row['user_id']}\t{tags}\t{ts}\t{op}"
    return hashlib.sha256(row_str.encode()).hexdigest()

for chunk in chunk_iter:
    hashes = chunk.apply(normalize_and_hash, axis=1).dropna()
    before_hashes.update(hashes)
```

`dtype=str`で型推論は回避できたのですが、数千万行に対する`apply`（内部的にはPythonのforループ）での文字列分割・ソート処理が致命的に遅く、メモリ枯渇に至りました。

### Polars：高速な新星

Rust製ライブラリのPolarsも試してみました。Lazy APIによるメモリ最適化に期待していました。

```python
import polars as pl

def pl_normalize(col_name):
    return pl.col(col_name).str.split("|").list.sort().list.join("|")

before_lazy = pl.scan_csv(
    "before/*.tsv", separator="\t",
    has_header=False, infer_schema_length=0
).filter(pl.col("column_4") != "D") \
 .with_columns([
    pl_normalize("column_2"),
    pl_normalize("column_3"),
    pl.when(pl.col("column_4") == "I")
      .then(pl.lit("U"))
      .otherwise(pl.col("column_4")).alias("column_4")
])
```

こちらは一部環境でバイナリ互換性に起因する実行エラーが発生することがあり、このエラーに躓いたため、ローカル検証時は使用を見送りました。

### DuckDB：安定した動作

DuckDBを試してみたところ、安定して処理が完遂しました。DuckDBが魅力に感じた点は、単一の性能指標ではなく、以下の4つの特性の組み合わせです。

1. **型制御の明快さ**: `all_varchar=True`を指定するだけで、すべての型推論を無効化し、純粋な文字列としてデータをロードできます。
2. **Out-of-Core実行**: メモリに乗り切らないJOIN処理でも、自動的にディスクへ一時退避しながら処理を完遂します。設定不要でOOMを回避できます。
3. **SQL完結性**: 複雑な正規化処理をSQLマクロとして記述でき、PythonレイヤーとCレイヤーの間のデータ変換が不要です。
4. **再現性**: `pip install duckdb`だけで動作し、Java環境もRustバイナリの互換性問題もありません。

以下のように実装することでうまく検証することができました。

```python
import duckdb

con = duckdb.connect()

# 順不同リストを正規化するSQLマクロ
con.execute("""
    CREATE MACRO normalize_pipe(s) AS
    list_string_agg(list_sort(str_split(s, '|')), '|')
""")

# 比較ロジックを1つのSQLで完結
# all_varchar=True が型推論の罠を防ぐキーポイント
query = """
    WITH
    before_data AS (
        SELECT
            column0 as user_id,
            normalize_pipe(column1) as tags,
            normalize_pipe(column2) as timestamps,
            CASE WHEN column3 = 'I' THEN 'U' ELSE column3 END as op
        FROM read_csv('before/*.tsv',
             delim='\t', header=False, all_varchar=True)
        WHERE column3 != 'D'
    ),
    after_data AS (
        SELECT
            column0 as user_id,
            normalize_pipe(column1) as tags,
            normalize_pipe(column2) as timestamps,
            column3 as op
        FROM read_csv('after/*.tsv',
             delim='\t', header=False, all_varchar=True)
    )
    SELECT
        COALESCE(b.user_id, a.user_id) as user_id,
        b.tags as b_tags, a.tags as a_tags,
        b.timestamps as b_ts, a.timestamps as a_ts
    FROM before_data b
    FULL OUTER JOIN after_data a ON b.user_id = a.user_id
    WHERE
        b.user_id IS NULL OR a.user_id IS NULL
        OR b.tags != a.tags
        OR b.timestamps != a.timestamps
        OR b.op != a.op
"""

# 差分結果をストリーミングでCSVに出力
con.execute(f"COPY ({query}) TO 'diff_report.csv' (HEADER, DELIMITER ',')")
```

### 標準実装の結果

まず、各エンジンでDataFrameをPython辞書に変換し、共通の正規化・差分計算ロジックを適用する「標準実装」の結果です。当初は複数エンジンを同時に実行しようとしたのですが、OOM Killerが発動してしまい（Pythonプロセスが約27.5GBを消費）、各エンジンを個別に順次実行する方針に切り替えました。

| エンジン | 実行時間 | メモリ使用量 | 差分行数 | 状態 |
| --- | --- | --- | --- | --- |
| Polars | 3分37秒（210秒） | 28.10 GB | 6,693,735 | 成功・最速 |
| Pandas | 4分59秒（294秒） | 23.83 GB | 6,693,735 | 成功・最省メモリ |
| DuckDB | 5分16秒（310秒） | 27.98 GB | 6,693,735 | 成功 |

全エンジンで同一の差分行数6,693,735行、出力ファイルサイズ147MBを確認しました。

ここで注目していただきたいのは、**標準実装ではPolarsが最速であり、Pandasが最も省メモリだった**という点です。最近メジャーバージョンアップしたPandasが意外と優秀でした。しかし、DuckDBは実行時間・メモリともに最下位でした。この結果は少し意外に感じたのですが、調べてみると標準実装がエンジンのネイティブ機能を活用できていないことに原因がありました。

### 標準実装のボトルネック

すべての標準実装は以下のパターンを使用していました。

```python
# 1. DataFrameをPython辞書リストに変換（2-3倍メモリ）
records = df.to_dict(orient="records")

# 2. 正規化処理で新しいリストを作成（2倍メモリ）
normalized = normalize_rows(records, config)

# 3. 差分計算で2つのマップを作成（2倍メモリ）
before_map = {make_key(r): r for r in before_normalized}
after_map = {make_key(r): r for r in after_normalized}
```

この実装では、データサイズの10〜15倍のメモリを消費してしまいます。2.6GBのデータに対して約28GBのメモリが必要だったのは、この変換処理が原因でした。つまり、どのエンジンを使っても「Pythonの辞書に変換する」という共通処理がボトルネックになっており、エンジン固有の強みが発揮されていない状態だったのです。

### 最適化実装の結果

標準実装の問題を踏まえて、各エンジンのネイティブ機能を活用する最適化実装を試してみました。「データをエンジン内に保持し、Python変換を排除する」というのが基本的な方針です。

| エンジン | 実装 | 実行時間 | メモリ使用量 | 状態 |
| --- | --- | --- | --- | --- |
| DuckDB | 標準 | 310秒 | 27.98 GB | 成功 |
| DuckDB | 最適化 | 4.2秒 | 1.04 GB | 成功 |
| Polars | 標準 | 210秒 | 28.10 GB | 成功 |
| Polars | 最適化 | 11.2秒 | 6.6 GB | 成功 |
| Pandas | 標準 | 294秒 | 23.83 GB | 成功 |
| Pandas | 最適化 | 20分超で強制終了 | 1.0 GB | 失敗 |

#### DuckDB最適化

SQL内でFULL OUTER JOINによる差分計算を完結させ、Python辞書変換を排除しました。結果として、310秒が4.2秒に、27.98GBが1.04GBになりました。データがDuckDBのベクトル化実行エンジン内で処理され、Pythonレイヤーへの変換が発生しなくなったことが改善の主因です。ただし、この最適化実装は正規化ロジックを簡略化しているので、標準実装と同一の処理ではありません。あくまで「Python変換を排除した場合にどの程度の改善余地があるか」を示す参考値として見ていただければと思います。

#### Polars最適化

Lazy評価（`scan_csv()`）とanti\_join/inner\_joinを組み合わせ、単一の`.collect()`で実行しました。210秒が11.2秒に、28.10GBが6.6GBに改善しています。

#### Pandas最適化の失敗

チャンク処理（100,000行ずつ）とハッシュベースの比較を実装しました。メモリ効率は1.0GBと良好だったのですが、`iterrows()`を使用していたため致命的に低速でした。

```python
# この実装が致命的に遅い
for idx, row in chunk.iterrows():
    key = row['__key__']
    before_index[key] = hash(tuple(row.drop('__key__').items()))
```

`iterrows()`は各行をSeriesオブジェクトに変換するため、1,500万行に対して約15,000秒（4時間超）を要する計算になります。`iterrows()`の処理速度は約1,000行/秒であるのに対し、ベクトル化操作は約1,000,000行/秒です。20分経過した時点で強制終了しました。Pandasのドキュメントでも`iterrows()`は非推奨とされているのですが、改めて大規模データでは使えないなと実感しました。

## 型推論の罠 — 一致検証における最重要課題

ここからは、この検証を通じて私が最も重要だなと感じた「型推論」の問題について掘り下げたいと思います。

### なぜ型推論が危険なのか

モダンなデータ処理ツールの型推論機能は、通常の分析作業ではとても便利です。ところが、「ファイル出力が文字列レベルで一致しているか」を検証する場面では、この便利さが正反対の結果をもたらします。

型推論が有効な状態で`"0123"`を読み込むと、ツールはこれを整数`123`として解釈します。新旧システムの出力が`"0123"`と`"123"`であった場合、型推論後はどちらも`123`となり、一致と判定されてしまいます。しかし、ファイル移行の検証では、前ゼロの消失は「バグ」として検知すべき差異です。

日時フォーマットについても同様です。`"2026-01-07 09:00:00"`と`"2026-01-07T00:00+0900"`はTIMESTAMP型として同一の時刻を指しますが、文字列表記としては異なります。移行テストではこの差異を検出しなければなりません。

### 各ライブラリの型推論制御

では、各ライブラリでどのように型推論を無効化できるのかを整理してみます。

| ライブラリ | 型推論無効化の方法 | 評価 |
| --- | --- | --- |
| Pandas | `dtype=str`を全列に指定 | 列が多い場合に煩雑 |
| Polars | `infer_schema_length=0` | 簡潔だが挙動の理解が必要 |
| DuckDB | `all_varchar=True` | 1つのオプションで全列が文字列 |

DuckDBの`all_varchar=True`が良いなと思ったのは、「型推論を無効化する」という意図がコード上で明確に表現される点です。一致検証においては、データの「意味」ではなく「表記」を比較することが目的です。この設計意図をコードに直接反映できることは、保守性の面でも大きな利点だと感じました。

### 文字列比較の原則

この検証を通じて、以下の原則が重要だなと強く感じました。

> データベース移行テストでは、データの意味（論理値）ではなく、出力されたフォーマット（物理値）の検証を優先すべきである。

型推論は「便利な機能」ですが、一致検証の文脈では「勝手にデータを加工する機能」になってしまいます。検証対象そのものが変質してしまうので、明示的に無効化する必要があります。

## 技術的学び — 性能比較を超えて

### 一致検証は設計問題である

本稿の結論は「DuckDBが最速だった」ではありません。実際、標準実装ではPolarsが最速でした。

重要なのは、一致検証の要件を整理し、それに適した技術を選定するプロセスだと思っています。今回の要件は、順不同リストの正規化、型推論の完全な無効化、数十GBデータの安定した処理、環境をまたいだ再現性の4点でした。これらを総合的に満たしたのがDuckDBであり、単一の性能指標で選んだわけではありません。

### 「Python変換」が最大のボトルネック

EC2での検証で最も明確になったのは、「エンジンからPython辞書への変換」がボトルネックの本質であるという点でした。

標準実装では、全エンジンが200〜310秒、23〜28GBのメモリを消費していました。エンジン間の差異よりも、共通のPython変換処理のコストの方がはるかに大きかったことになります。最適化実装でPython変換を排除した結果、DuckDBは4.2秒・1.04GB、Polarsは11.2秒・6.6GBまで改善しました。

この事実から、「エンジン選定」以上に「エンジンの使い方」が性能を決定するのだなと実感しました。

### Out-of-Coreの実用的価値

ローカル環境（メモリ36GB）でもEC2環境（メモリ32GB）でも、データサイズの10倍以上のメモリを消費する処理が発生しました。EC2ではOOM Killerが発動し、プロセスが強制終了されています。

DuckDBのOut-of-Core実行は、メモリ不足時に自動的にディスクを利用するため、設定不要でこの問題を回避できます。最適化実装では1.04GBしか使用せず、32GBのインスタンスで余裕を持って動作しました。処理対象のデータサイズが事前に読めない状況では、この特性は実用上とても心強いです。

### 再現性の確保

今回の検証では、CloudFormationによるEC2環境の自動構築、S3経由でのコード配布、SSM Session Managerによるセキュアな接続など、再現可能なインフラ構成を採用しました。検証結果の信頼性は、コードだけでなく実行環境の再現性にも依存するので、この辺りも気を配りました。

DuckDB自体の導入コストの低さ（`pip install duckdb`のみ）も、再現性確保に貢献しています。Java環境の構築やRustバイナリの互換性を気にする必要がないので、ローカルとEC2のどちらでも同じ手順ですぐに検証を始められたのは実際に助かりました。

## ベンチマーク結果サマリー

最後に、実測値をまとめます。

## まとめ

数十GB規模のデータ移行検証において、「よしなにやってくれる便利な機能（型推論）」は、時に深刻な誤判定の原因になるということを身をもって学びました。一致検証は、適切な型制御の設計、エンジンのネイティブ機能の活用、そして再現可能な環境構築の3つが揃って初めて信頼できる結果を得られるのだなと感じています。

本稿を通じて伝えたかったことを3点にまとめます。

1. **文字列の検証は、文字列のまま行う** — データの意味（論理値）ではなく、出力されたフォーマット（物理値）の検証を優先すべきです。
2. **エンジンの使い方が性能を決める** — Python変換の排除は、エンジン選定以上に大きな効果をもたらします。
3. **ローカルでの大規模処理にはDuckDB** — Pandasで`MemoryError`が出たとき、すぐにSparkクラスターを立ち上げるのではなく、まずは手元のDuckDBにSQLを投げ込んでみてください。

本稿で紹介した試行錯誤の記録が、データ移行検証を行なっている方の一助となれば幸いです。

## 最後に

ローカルのPCでギリギリ処理できない規模のデータを使用してどうやって一致検証をさせるか苦戦した内容をもとに、無風のEC2インスタンス環境を作成して検証したらどうなるかが気になったのでやってみました。今回は検証に使える時間が一日だけだったのでそこまで詳細に比較できていなかったですが、もっとインスタンスのタイプを小さいものにしてみたり、Pandasのバージョンを2系に変えた場合にどうなるか、というところもやってみたいです。
また、EC2インスタンスでは検証できましたが、Lambdaでできるかどうかも挑戦してみたいなと思っています。
色々とやりたいことが出てきてしまいましたが、限られた時間の中でも比較できて良かったです。
