# バージョン管理システム Jujutsu

Jujutsu（jj）は、近年注目を集めている新しいバージョン管理システムです。この記事では、Git を使ったことがある人向けに jj の特徴をいくつか紹介します。

## バージョン管理システムとは

バージョン管理システムとは、ファイルの変更履歴を記録・管理するためのシステムです。ソフトウェア開発において、ソースコードの変更を追跡し、過去の状態に戻したり、複数人で同じコードベースを編集したりすることを可能にします。

現在もっとも広く使われているバージョン管理システムは Git です。Git は 2005 年に最初のバージョンがリリースされました。その後 GitHub が 2008 年に登場し、2010 年代の前半には Git や GitHub が広く普及していきました。今から 10 年前にはすでに Git がデファクトスタンダードとなっていました。そのため、今となっては Git 以外のバージョン管理システムを使ったことがないという人も多いでしょう。

## Jujutsu とは

Jujutsu は、2022 年に公開が始まった新しいバージョン管理システムです。公式サイトは <https://www.jj-vcs.dev> です。この記事の執筆時点（2026 年 2 月）での最新版は 0.38.0 です。バージョンが 0 始まりで、まだ安定版としてはリリースされていません。しかし、2024 年ごろから技術系のコミュニティで注目が高まっています。

なお、ツールやコマンドの jj という名前が先に誕生して、それにマッチする名称として Jujutsu という単語が選ばれたという命名の経緯があります。「柔術」とも「呪術」とも捉えられそうな単語ですが、具体的にどの言葉を指しているのかは公式では決まっていないそうです。この記事では Jujutsu のことを単に jj と呼ぶことにします。

jj はバックエンドに Git を利用しています。Git と併用可能であり、Git からの移行の手間が少なく使い始めやすいです。なお、将来はバックエンドに Git 以外を使うことも検討されています。

## なぜ jj を使うのか

jj は新しいバージョン管理システムですが、実のところ、jj でできることはほとんど Git でもできます。むしろ、Git のほうが多機能と言っていいでしょう。では、なぜ jj を使うのでしょうか。

jj は、Git とは異なる独自の概念やインタフェースを採用しています。その概念がうまくできていて、バージョン管理システムで行いたいことが、よりストレスなく実行できます。これが大きな点で、jj は日々の開発における作業を快適にしてくれます。

では、実際に jj を使ってみましょう。jj のインストールについては公式サイトをご覧ください。

## jj リポジトリを作る

まずはリポジトリの作り方です。管理したいフォルダで `jj git init` コマンドを実行します。これにより `.jj` フォルダが作成されます。

```console
❯ jj git init
```

また、既存の Git リポジトリを jj リポジトリにできます。それには Git リポジトリで `jj git init` を実行すればよいです。`.jj` フォルダが作成される一方で `.git` フォルダもそのまま残ります。`jj` コマンドと `git` コマンドのどちらも利用できる状態となります。

この jj と Git のハイブリッド状態は colocated と呼ばれています。なお、init 時のオプションで `--no-colocate` をつけることで、`.git` を見えなくして `.jj` のみにもできます。

## jj のチェンジとは

リポジトリ管理下の作業コピー（Working copy）の状態を確認するには `jj status` コマンドを使います。

```console
❯ jj status

The working copy has no changes.
Working copy  (@) : qtnqlkkm 6f10bcb1 (empty) (no description set)
Parent commit (@-): vtlvuklt e408ab58 コミットメッセージ
```

この出力例を見ながら、jj の概念を説明します。

jj の重要な概念のひとつが「チェンジ（Change）」です。Git では「コミット」が変更の単位ですが、それに相当するものが jj では「チェンジ」です。

まんなかのあたりに、`qtnqlkkm 6f10bcb1` や `vtlvuklt e408ab58` という表示があります。この各行にある ID の並びは jj のチェンジをあらわしています。jj のチェンジはふたつの ID で表現され、前半の `qtnqlkkm` や `vtlvuklt` はチェンジ ID、後半の `6f10bcb1` や `e408ab58` はコミット ID と呼びます。なお、チェンジ ID には `k-z` の 16 種類の文字が、コミット ID には `0-9a-f` の 16 種類の文字が使われます。これを利用すると、使われている文字を見れば ID の種類が区別できます。

Git では、コミットに対して `git commit --amend` でコミットの内容を変更したり、rebase などで履歴を変更したりすると、コミット ID が変わり別のコミットとなります。一方 jj では、チェンジの内容を変更したり履歴を変更したりしても、チェンジ ID は変わらず同じチェンジとなります。jj のチェンジはある変更作業をあらわす論理的なまとまりです。Git のコミットが変更の実体を表現しているのとは異なっています。

jj のチェンジはふたつの ID で表現され、チェンジの内容を変更してもチェンジ ID は変わらないと述べました。一方、そのチェンジのコミット ID は変わります。これは、同じ作業のまとまりの中でも、内容には更新があることをあらわしています。

## 作業コピーもチェンジである

Git では、作業コピーで何か変更したあと、`git add` をすることではじめてインデックスに追加され、`git commit` をすることでインデックスからコミットが作成されます。

jj では、作業コピーもチェンジのひとつとして扱われます。自動的にチェンジ ID が作成されており、作業コピーで何か変更があると `jj` コマンドがそれを認識してチェンジに自動的に反映します。

こう聞くと、Git に慣れている身からすると懸念が生じます。Git のインデックスは、望むコミットを整える場として有益なものです。それがなくなって、勝手にコミットされてしまうのでは、使い勝手が悪くなっていないか、と。

実際に使ってみるとこの点は大丈夫です。Git のインデックスを使って行っていた作業は jj でも問題なく行えます。jj のチェンジは柔軟な操作が可能であり、さまざまな作業フローが実現できます。

## チェンジの説明を書く

作業コピーで何か変更をした場合の `jj status` の出力例です。この例では `README.md` ファイルが変更されていることが表示されています。

```console
Working copy changes:
M README.md
Working copy  (@) : qtnqlkkm 382665cb (no description set)
Parent commit (@-): vtlvuklt e408ab58 コミットメッセージ
```

これは先ほど `jj status` で見た状況の続きです。作業コピーのチェンジ ID は `qtnqlkkm` で変わらず、コミット ID は `382665cb` と変わっています。

さて、チェンジ ID やコミット ID の後ろに `(no description set)` と書かれています。チェンジには説明（description）をつけることができます。やってみましょう。

`jj describe` コマンドを実行するとエディタが起動しますので、説明のテキストを入力すればよいです（またはコマンドの `-m` オプションでテキストを渡すこともできます）。

```console
❯ jj describe

Working copy  (@) now at: qtnqlkkm 345b8287 README を更新
Parent commit (@-)      : vtlvuklt e408ab58 コミットメッセージ
```

`jj describe` は好きなタイミングで実行できます。作業コピー上での作業開始前でも、作業の途中でも構いません。あとで説明を書き換えることもできます。また、説明を書かないまま先に進むこともできます。

ただ、あとで Git リポジトリにプッシュしたくなった場合には、説明のないチェンジが存在しているとプッシュできません。この説明が Git のコミットメッセージとなるためです。

## 新しいチェンジを作る

ここで変更を一区切りにしたいとしましょう。jj では、変更はすでにチェンジに反映されているので、Git でいう `git commit` 操作はしません。代わりに、新しいチェンジを作る操作をします。

新しいチェンジを作成するには `jj new` コマンドを使います。

```console
❯ jj new

Working copy  (@) now at: vrlsmwrz 34cc4ec2 (empty) (no description set)
Parent commit (@-)      : qtnqlkkm 345b8287 README を更新
```

先ほどまで作業コピーだったチェンジ `qtnqlkkm` の代わりに新しいチェンジ `vrlsmwrz` ができました。次の作業コピーの変更は新しいチェンジに反映されます。

ところで、Git に慣れている身からすると、ただコミットしたいだけで次の作業予定がないときでも「新しいチェンジを作る」ということに抵抗を感じてしまいます。

ここで思い出したいのが、「作業コピーもチェンジである」ということです。何も新しい作業を始めるときではなくとも、単に作業コピーを用意するだけのときにも、jj ではそれをチェンジで表現するので新しいチェンジを作るのです。ここは発想の転換がいります。

そして、何の変更もなく（`(empty)`）説明のテキストもない（`(no description set)`）空のチェンジは、不要になれば `jj` コマンドが自動的に削除します。そのため、気軽に新しいチェンジを作っても構いません。

## ログを確認する

チェンジの概念を知り、`jj describe` と `jj new` を知れば、簡単なことであれば jj を実際に使っていくことができます。ここから、さらに実践的なことを知っていきましょう。

ここまでの変更履歴を確認してみましょう。`jj log` コマンドを使います。

```console
❯ jj log

@  vrlsmwrz dummy@example.com YYYY-MM-DD HH:MM:SS 3a48ad63
│  (empty) (no description set)
○  qtnqlkkm dummy@example.com YYYY-MM-DD HH:MM:SS 345b8287
│  README を更新
○  vtlvuklt dummy@example.com YYYY-MM-DD HH:MM:SS e408ab58
│  コミットメッセージ
◆  zzzzzzzz root() 00000000
```

それぞれのチェンジの内容を確認するには `jj show` コマンドを使います。引数にはチェンジ ID を渡します。

```console
❯ jj show qt

Commit ID: 345b8287234de6bceba21f5186d39c5448e3470d
Change ID: qtnqlkkmuomnvyxmnmrsqmzsqpnkpwts

（・・・チェンジの内容・・・）
```

チェンジ ID は最初の数文字を指定すればよいです。また、`@` で現在の作業コピーを指定できます。`@-` と後ろに `-` をつけると親のチェンジを指定できます。

チェンジ ID の指定自体を省略した場合は `@` を指定したのと同等になります。つまり、単に `jj show` とすれば現在の作業コピーのチェンジとしての内容を確認できます。

なお、さらに細かいログも確認できますので、簡単に触れておきます。`jj operation log` コマンドを使うと、`jj` コマンドの実行履歴が得られます。そして、`jj undo` コマンドを使うと、その操作を取り消すことができます。

## チェンジの操作いろいろ

jj のチェンジは柔軟に操作できます。ここでは、そのいくつかを紹介します。

### チェンジを分割する

`jj split` はチェンジを分割します。作業コピーに変更がある状態で `jj split` を実行すると、次のような TUI の画面が出てきます。

```console
[File] [Edit] [Select] [View]
[ ] README.md                         [+]
[ ] doc.md                            [+]
```

TUI 画面の主な操作は次のとおりです。分からなくなったら `?` でヘルプを見ましょう。

* `?` で操作ヘルプを表示
* `q` で中断して終了
* `c` で決定して終了
* スペースキーで選択

現在のチェンジに残したい変更内容を選択して `c` で終了すると、新しいチェンジが作成されて、選択されなかった内容が新しいチェンジに移動します。

つまり `jj split` は、Git の `git add -p`（内容を指定してインデックスに追加する）に似たようなことができるわけです。

### チェンジ内の変更内容を移動する

`jj squash` は現在のチェンジでの変更内容を親のチェンジに移動します。また `jj squash -i` と `-i` オプションをつけると、移動する内容を選択できます。選択画面は先ほどの TUI 画面と同様です。

たとえば `jj split` で新しいチェンジを作成して分離した内容があるとします。それを再度親のチェンジに入れる、ということが簡単にできるわけです。

### チェンジ内の変更内容を元に戻す

`jj restore` は現在のチェンジでの変更内容を破棄します。また `jj restore -i` と `-i` オプションをつけると、破棄する内容を選択できます。選択画面は先ほどの TUI 画面と同様です。

Git の `git restore` と同じようなことができるわけです。

### チェンジを削除する

`jj abandon` は指定したチェンジを削除します。

何の変更もなく説明のテキストもない空のチェンジは自動的に削除されるのでした。しかし、何か変更があったり説明のテキストを書いていたりすると自動的には削除されません。不要になったチェンジはこのコマンドで明示的に削除します。

### 空のチェンジを挿入する

`jj new -B @` と `-B` オプションを使うことで、現在のチェンジの前に空のチェンジを挿入できます。

`jj new -B @` によって空のチェンジを挿入したとき、作業コピーは新しく挿入されたチェンジになります。`jj log` を見ると `@` のマークの位置が現在の作業コピーです。

```console
○  vrlsmwrz dummy@example.com YYYY-MM-DD HH:MM:SS 297c869e
│  (no description set)
@  xwnqswos dummy@example.com YYYY-MM-DD HH:MM:SS 8ca7a549
│  (empty) (no description set)
○  qtnqlkkm dummy@example.com YYYY-MM-DD HH:MM:SS 345b8287
│  README を更新
```

このまま作業すれば、チェンジ ID `xwnqswos` のチェンジの変更になります。

`jj new -B @` の前の作業コピーを変更する状態に戻るには `jj edit` コマンドを使います。いまの場合だと、`jj edit vr` とすれば作業コピーを `vrlsmwrz` のチェンジに戻すことができます。

### ブランチを作る

`jj new` でチェンジ ID を指定すると、指定のチェンジからのブランチを作成できます。

Git では、ブランチの作成ではブランチ名をつけていました。しかし、jj では匿名ブランチが基本です。名前を考える必要がないので気軽に作成できます。

また、この操作は作業コピーの切り替えでもあります。特定のチェンジの状態に作業コピーを切り替えるというのは、実はその特定のチェンジから匿名ブランチを作ることと同じだと考えることができます。

### ブランチをマージする

ブランチをマージするには、`jj new` でチェンジ ID を複数指定します。

Git では、マージの際にはコンフリクトが起こればそれを解消する必要がありました。しかし、jj ではコンフリクトを解消しないままマージできます。

もちろん、どこかのタイミングでコンフリクトは解消しなくてはなりません。ただ、それはマージのときでなくてもよいのです。コンフリクトを含んだままマージをして、さらにその先のチェンジも進めることができます。このコンフリクト解消の先送りは、作業のストレス軽減になります。

コンフリクトを解消するには、そのチェンジに `jj edit` で移動して、対象ファイルを直接編集すればよいです。コンフリクトが発生している箇所はマーカーが挿入されていますので、その部分を編集します。あるいは `jj resolve` コマンドを使うこともできます。

### 履歴ツリーを操作する

`jj rebase` はチェンジの履歴ツリーを操作できます。Git のリベースよりも強力で、どのチェンジをどこに移動するか柔軟に指定できます。

どこに移動するかというのは、指定したチェンジの上に乗せるだけでなく、チェンジとチェンジの間に割り込ませる、ということもできます。これを応用して、チェンジの順序を入れ替えたりもできます。

### その他のコマンド

jj のコマンドはたくさんあります。CLI Reference <https://www.jj-vcs.dev/latest/cli-reference/> を参照してください。たとえば NotebookLM に読み込ませると便利です。
